# Lite Mode 需求分析

## 1. 背景与定位

tight 是面向**端云实时通信场景（IoT 设备 ↔ 云端网关）**的自研可靠 UDP 传输协议栈
（`tight/README.md:3`、`tight/docs/usage.md:3`）。协议需要同时运行在两类硬件条件悬殊的节点上：

- **云端网关（Node）**：多核服务器，需承载成百上千并发连接、吸收多设备突发流量；
- **终端设备（Leaf）**：嵌入式 IoT 设备（音频对讲、传感器、摄像头），RAM 有限、CPU 单核为主、
  电池供电或对功耗敏感、通常只维护**单连接**。

lite mode（客户端精简模式）即为终端设备侧的运行模式
（`types.hpp:123-127`："客户端精简模式：单线程……面向嵌入式/单连接客户端"）。

## 2. 目标用户与典型场景

来自 `tight/docs/usage.md` 第 9 节的三个典型配方：

| 场景 | 业务特征 | 关键配置 |
|---|---|---|
| 9.1 IoT 音频设备 | 16kHz G.711/PCM 音频 + 视频 + 偶尔文件传输，实时性优先 | `lite_mode=true`、mtu=1350（恰容纳 40ms PCM 帧 1280B）、`retransmit_enabled=false` 纯实时 AV |
| 9.2 云端网关 | 多设备汇聚、突发吸收 | **保持普通 4 线程**、`queue_limit=4096`、socket 缓冲 8MB |
| 9.3 低速遥测 | 极致省电、间歇小数据 | `lite_mode=true`、关测速、`max_message_bytes=8KB`、心跳 10s |

角色模型：`LinkRole::Node`（服务器接受接入）/ `LinkRole::Leaf`（终端设备）（`types.hpp:44-47`）。

## 3. 功能性需求

| 编号 | 需求 | 说明 |
|---|---|---|
| FR-1 | 协议行为两端一致 | lite 是本端属性：客户端 lite、服务器普通模式互不影响；握手/加密/测速等协议行为完全一致（`usage.md:400-401`） |
| FR-2 | 运行时切换 | `set_lite_mode()` 在 `start()` 前后均可调用，安全切换线程模型（`transport.cpp:157-209`） |
| FR-3 | 可靠传输完整能力 | lite 模式下 ACK/NACK 重传、RS FEC、乱序重组、命令通道、加密、测速等能力不缺失 |
| FR-4 | 单连接单线程 | 终端通常一条连接，允许以单线程 reactor 合并收发/编码/发送职责 |
| FR-5 | 静默降级 | 资源耗尽（队列满、重组超限）时静默丢弃，不产生日志 I/O 冲击（lite 强制关 drop_log） |

## 4. 非功能性需求

### 4.1 内存（核心约束）

| 指标 | 目标 | 验证手段 |
|---|---|---|
| 空闲实例驻留 | **~76KB**（`tight.hpp:47-51`、`usage.md:402`） | `GetProcessMemoryInfo` 测量 delta |
| 传输期驻留 | ≈ 在途字节数，与总流量无关（平台期） | 16MB 传输 delta ~136KB 平台期（`usage.md:402-403`） |
| 无泄漏 | 多轮收发曲线呈平台期（CRT 高水位保留属正常） | `tests/test_mem8k` |
| 线程栈 | 单线程 64KB 栈（系统默认 1MB 的 1/16） | `SmallThread`（`small_thread.hpp:62-82`） |

### 4.2 功耗 / CPU

- flush 节拍钳制 **≥10ms**（普通模式可低至 1-2ms）：唤醒频率 500 次/s → 100 次/s，
  附加延迟 ≤10ms（`types.hpp:87-89`、`transport.cpp:147-153`）。
- 单线程避免多线程调度与同步开销。

### 4.3 实时性

- 音频帧端到端延迟优先于可靠性：`retransmit_enabled=false` 时在途内存常数化（~24KB）；
- mtu=1350 → 单帧单包，无分片重组延迟。

### 4.4 可移植性

- 仅依赖 C++17 标准库 + 系统 socket API（Winsock2 / POSIX）；加密算法纯 C++ 自实现零依赖，
  便于交叉编译到嵌入式工具链。

## 5. 约束与权衡

| 约束 | 设计决策 |
|---|---|
| 队列容量构造期固定 | encode/outbound 队列容量在 ctor 即定型，运行时 `set_lite_mode` 只切线程模型——保证切换安全性与内存预算确定性（`transport.cpp:155-156` 注释） |
| socket 内核缓冲 ≤16KB | lite 无汇聚突发场景，16KB 够用；网关侧必须用普通模式 + 8MB 缓冲，否则突发静默丢包（`transport.cpp:296-303` 注释） |
| 重传缓冲 vs 内存 | 可靠性可协商关闭：关闭侧不生成 NACK、缺口立即跳过，在途内存不随码率增长（`report.cpp:46-52`） |
| 数据面洪泛必丢 | UDP 内核缓冲溢出不可避免 → 文件类传输需应用层块校验 + 补发（协议不变量） |

## 6. 验收标准

1. 13/13 ctest 套件全绿（含加密向量、sim_device 端到端）；
2. `test_mem8k` lite 客户端：进程地板 ~5.5MB、tight 实例 ~76KB、50KB/s×8KB 稳态 delta ~136KB 平台期；
3. lite ↔ 普通模式互通（协议行为一致性）；
4. 运行时 `set_lite_mode` 往返切换不丢包、不死锁、无在途报文滞留（`m_lite_pending` 槽位处理）。
